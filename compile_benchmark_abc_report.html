Where the improvements come from

A → B (−4.9 s, −13%): front-end and middle-end optimizations

These changes live in the vendored vendor/artiq and vendor/ndscan trees and reduce the work done on every compile, cached or not:

  • Dirty-set type inference (artiq/compiler/embedding.py): the embedding fixed-point loop tracks per-function typed-AST hashes and re-visits only functions whose types actually changed, instead of sweeping all functions every round. Measured on this fragment: 4,789 function visits (32 discovery cascades + 17 hash rounds) vs upstream's ~18 full passes × 861 functions ≈ 15,500 visits — a 3.2× reduction in inference work. It reaches the same fixed point; correctness was regression-tested across all LMT and a sample of non-LMT fragments.
  • Per-(value, attribute) interrogation cache: _unify_attribute no longer re-loops over all host values on every visit of every attribute node (was ~55k redundant calls).
  • Visitor dispatch + type-hash caching: resolved visit_* methods are cached per (visitor, node) class pair — the compiler visits millions of AST nodes per LMT compile — and typed-tree hashes are cached within a hash round.
  • ndscan forwarder deduplication: kernel_from_string results for the synthesized setup/cleanup forwarders are shared per class instead of regenerated per fragment instance, shrinking the embedded set from 861 to 742 functions. Fewer functions means less work in every later phase (ARTIQ IR, LLVM IR, codegen) too.

B → C (−5.0 s, −15%): content-addressed kernel cache

The branch adds artiq.compiler.kernel_cache: after LLVM IR generation, the IR text plus an environment fingerprint (compiler-source digest, llvmlite/LLVM versions, target/linker options) is hashed with blake2b; on a hit the linked ELF is loaded from disk and LLVM parse, verify, optimize, codegen and link are skipped (15.4 s → 11.0 s backend in the diagnostic runs). Host attribute values are embedded in the IR as global initializers, so the key covers values as well as code — a stale hit is impossible.

Making this work cross-process was the hard part: a content-addressed cache requires byte-identical IR for identical input, and three sources of memory-address-dependent nondeterminism had to be fixed (string-kernel names derived from id(), set-ordered constant hoisting, set-ordered exception-phi predecessor choice). This benchmark exercised that path for real — every warm hit came from a cache written by a different process, with identical code hashes.

Why a warm hit is 28 s and not 3 s

The cache key is the LLVM IR text, so everything upstream of it — inference (12.8 s), ARTIQ IR (4.8 s) and LLVM IR generation (10.9 s) — must still run on every compile just to compute the key. Skipping those phases would require a key computable before inference (covering the transitive closure of kernel sources and all embedded host values) plus separate upload of parameter values to the device; the core device protocol has no memory-write command today, so that is firmware + protocol work, not a compiler patch.

Caveats and deployment notes

  • pyaion blocker for production caching: UrukulInit.host_setup embeds hash(device) (a memory address) into kernel data and orders CPLDs via list(set(...)), so any Urukul-touching experiment compiles to different bytes per process. The test suite patches this in tests/conftest.py; the same one-method fix is needed in pyaion before live re-runs hit the cache.
  • Parameter-value changes are cache misses: values are baked into the IR globals, so re-running the same experiment with the same values hits, but changing any embedded host value recompiles the backend. (Scanned ndscan parameters are uploaded at runtime and do not invalidate.)
  • The nix dev shell still pins pristine artiq/ndscan; the optimized trees are exercised via sys.path injection (scripts/pytest_vendored.py). Deployment means pushing the vendored changes to the aion-physics forks and updating flake.lock.
  • Leg A is the pinned baseline the branch forked from (aion-physics artiq fork @ e98dbf5a, make-event-spreading-optional), i.e. exactly what production compiles with today.

Methodology

  • Machine: 4-core Intel Xeon @ 2.80 GHz, 15 GB RAM, Linux container; Python 3.10.12 from the project nix environment; branch at bf2c09dc.
  • Workload: single deepest-tree LMT compile test, tests/test_compile_all.py -k LMTInterferometryWithDoubleLaunchSingleImageFrag (compiles device_setup → run_once → device_cleanup as one kernel, as production does). All 9 runs passed.
  • Design: interleaved A→B→C, 3 iterations, fresh pytest process per run; warm directory populated once by a separate untimed run. Loaded module paths are asserted per run (vendored legs refuse to run against /nix/store packages and vice versa).
  • Phase breakdowns come from two additional instrumented diagnostic runs (ARTIQ_PHASE_TIMING=1, output capture disabled), not from the timed runs.
  • Raw logs: /tmp/bench_logs/ in the benchmark container.
