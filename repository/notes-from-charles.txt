Notes from charles to alice about spin polarization
===================================================

Hey, I've been looking through the code to remind myself why we have a problem.
The relevant code is all in the red_beam_controller module, in methods like
`turn_on_spin_pol` etc. This is currently disabled by line 45 in the same file,
where I have commented out the beam info that defines the spin pol beam.

The reason I did this is frustrating. The spin pol AOM is on the same
AD9910-Urukul as the injected diodes which makes sense because they both do not
require SUServo feedback and might need fast modulation. However there's a
*detail* of how the Urukuls work. You can read about this in the comments in
`glitchfree_urukul_default_attenuation` and in the labbook entry that they link
to. In summary, setting or even just reading the attenuator of any urukul
channel will cause a small glitch in every other channel on the same urukul.
This is enough to unlock the red IJD, which means that if we just use the
default beam setter like we do for every other beam (search for
`SetBeamsToDefaults` in the PyAION documentation (which is hosted on gitlab
pages - there's a link from the pyaion repository)) then this will happen every
time we run the experiment.

There's a good solution to this problem which would get rid of that glitch, but
this is too annoying for us to bother with - it involves changing the FPGA code
that's running on the Urukuls themselves, not even the Kasli.

To avoid this problem therefore, I already made
`glitchfree_urukul_default_attenuation` which is currently used for setting up
the red IJD AOM itself. Please read this code too, to understand how it works.

What you'll need to do is:

a)  Remove the spin polarization AOM from the default beam setter. It's already
    commented out, but you can just delete the line entirely since I think we won't
    use the beam setter for this one.

b)  Add a `GlitchFreeUrukulDefaultAttenuation` to the `red_beam_controller` which
    will set up the spin polarisation AOM's attenuator once only.

c)  Add code to `RedBeamController.init()` to initiate the spin pol AOM's
    frequency. It's attenuation will be static.

d)  Add a parameter to `RedBeamController` controlling the spin pol frequency.
    As a default value, you can use the existing `UrukuledBeam` information in
    `constants.URUKULED_BEAMS`. You should use this `UrukuledBeam` information
    to set the amplitude and attenuation too, but these don't need parameters
    (the attenuation can't have one because it will be ignored by
    `glitchfree_urukul_default_attenuation`, the amplitude is unnecessary since
    we already have SUServo setpoints to control that more accurately).

e)  Test it out! Initially use the picoscope to check that your code is working
    properly.

Testing it and using it
=======================

Currently, the only code which uses the spin pol beam is the
`DroppedPumpedLatticeMixin` in `pumped_lattice`. This would be a reasonable
place to try it out from when you're just testing the code to see if the light
turns on properly. You should already have the ability to:

a) Toggle between s+ and s- polarizations of the spin-pol light to choose which
stretched state you want.

b) Adjust the SUServo setpoint

Once you come to using atoms, I'd suggest re-reading the Ludlow thesis chapter
which talks about spin polarization to remind yourself how it's meant to work.
We have yet to actually use spin polarization.
You could try to use it with Sr87 by driving the clock transition with
`BasicClockSpecFromLatticeFrag`.

However, I think you might find it easier to set up if you instead do 689nm
spectroscopy on the Zeeman components. That would avoid the need for you to mess
around with the clock laser, which is one of the most annoying spinning plates
in the lab at the moment. It would also give you access to a much larger
differential Zeeman shift (since the 3P1 state is more sensitive than the 3P0),
so you should be able to resolve non-degenerate Zeeman transitions without
having to use long clock pulses (for their low Fourier width). As a bonus, it
would help us to measure our magnetic fields better.

The beam to use for this is the UP beam since it's the most intense. We have
messed with the SUServo since the previous Zeeman measurements were made so
you'll probably need to disable the SUServo and just set it to maximum. It also
used to have linear polarisation at the atoms, but has now been tuned to
circular (we think!), so do bear that in mind for your choice of polarisation.
If you use the Y bias coils for a Y quantization axis, then you'll still have
access to all three transitions.

To do this, you could try to replicate the dmF = +-1 measurements that were done
with Sr88 but in Sr87 instead, then see if you can state prepare atoms into a
stretched mF state to give you better fidelity. You could evaluate this simply
by measuring the number of atoms left in the 1P0 state after your 689nm pulse,
avoiding the need for fast kinetics mode and normalization to start with. You
should find that the `SpectroscopyWithKinetics_UpBeam` experiment is most of the
way there, although you'll need to make a copy of it which:

* adds spin polarisation
* adds lattice trapping (probably just use the dropped lattice mixin for both of these)
* changes the `TripleImageFastKineticsMixin` for a simpler `SingleAndorImage`.

I would suggest not dropping from the lattice, so just setting the high and low
power setpoints to the same value if you're using the
`DroppedPumpedLatticeMixin`.

That's a lot of stuff, much of which I expect you'll have questions about. You
may wish to start with just making an ExpFragment as a test, which does
something simple like setting an Urukul frequency according to the value of a
parameter so that you can get used to NDScan. Have an explore of the
`repository.tests` directory for inspiration - you'll find that these are a lot
simpler than the main code.

And feel free to text me questions! Though I can't promise to answer quickly.
